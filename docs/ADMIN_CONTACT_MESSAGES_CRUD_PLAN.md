# Admin Contact Messages CRUD Plan

## Goal

Implement a read-manage workflow for contact messages in the admin panel. The public contact form submission endpoint will also be built so the admin side has data to work with, but the public-facing contact page UI is deferred.

This plan covers:

-   database migration for `contact_messages`
-   backend entity, repository, service, DTOs
-   public `POST /api/contact` endpoint (unauthenticated submission)
-   admin CRUD endpoints for managing messages
-   admin list page with search and status filters
-   admin detail view (read-only, status update)
-   verification criteria

This plan does **not** include:

-   public contact page UI (placeholder already exists)
-   email notifications on new messages
-   reply-to-sender functionality
-   spam filtering or CAPTCHA

## Domain Model

A `ContactMessage` is a lightweight inbound message from a site visitor:

| Field     | Type            | Notes                                      |
| --------- | --------------- | ------------------------------------------ |
| id        | `Long`          | auto-generated                             |
| name      | `String(120)`   | required, trimmed                          |
| email     | `String(254)`   | required, valid email format               |
| subject   | `String(200)`   | required                                   |
| message   | `Text`          | required                                   |
| status    | `ContactStatus` | enum: `UNREAD`, `READ`, `ARCHIVED`         |
| createdAt | `Instant`       | auto-set                                   |
| updatedAt | `Instant`       | auto-set                                   |

### Status Lifecycle

```
UNREAD  →  READ  →  ARCHIVED
  ↑                   │
  └───────────────────┘ (reopen from archived)
```

-   New submissions default to `UNREAD`.
-   Admin marks as `READ` after viewing.
-   Admin can archive (`ARCHIVED`) to hide from the default list.
-   Admin can move back from `ARCHIVED` to `UNREAD` if needed.

### Business Rules

-   `POST /api/contact` is publicly accessible (unauthenticated).
-   All admin endpoints require authentication.
-   Admin list defaults to non-archived messages (`UNREAD` + `READ`).
-   Admin can filter by status.
-   Admin can search by name, email, or subject.
-   Admin can delete messages (hard delete).
-   Admin cannot create or edit messages — only the public can submit.
-   No PII beyond the visitor's name and email is stored.

## Proposed API Contract

### Public

```
POST /api/contact
  Request:  { name, email, subject, message }
  Response: 201 { id, name, email, subject, message, status, createdAt, updatedAt }
```

### Admin

```
GET    /api/admin/contact-messages?page=&size=&search=&status=
  Response: PagedResponse<ContactMessageResponse>

GET    /api/admin/contact-messages/{id}
  Response: ContactMessageResponse

PUT    /api/admin/contact-messages/{id}/status
  Request:  { status }
  Response: ContactMessageResponse

DELETE /api/admin/contact-messages/{id}
  Response: 204 No Content
```

### DTO Shapes

```java
// Request
record ContactMessageRequest(
    @NotBlank @Size(max = 120) String name,
    @NotBlank @Email @Size(max = 254) String email,
    @NotBlank @Size(max = 200) String subject,
    @NotBlank String message
) {}

// Response
record ContactMessageResponse(
    Long id,
    String name,
    String email,
    String subject,
    String message,
    ContactStatus status,
    Instant createdAt,
    Instant updatedAt
) {}

// Status Update
record ContactStatusUpdateRequest(
    @NotNull ContactStatus status
) {}
```

### Contract Rules

-   Public POST creates messages with `UNREAD` status — caller cannot set status.
-   Admin list is paged and sorted by `createdAt DESC, id DESC`.
-   Admin list defaults to `UNREAD` + `READ` when no status filter is provided.
-   Admin status update must validate the status enum.
-   Admin search matches against `name`, `email`, and `subject`.
-   Deletion is permanent (hard delete).
-   Validation failures use field-aware error responses.

## Database

### Migration

`V9__add_contact_messages.sql`:

```sql
CREATE CAST (varchar AS contact_status) WITH INOUT AS IMPLICIT;

CREATE TABLE contact_messages (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name       VARCHAR(120) NOT NULL,
    email      VARCHAR(254) NOT NULL,
    subject    VARCHAR(200) NOT NULL,
    message    TEXT         NOT NULL,
    status     contact_status NOT NULL DEFAULT 'UNREAD',
    created_at TIMESTAMP WITH TIME ZONE NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL
);
```

> `contact_status` is an existing PostgreSQL enum if the project uses enum types. If not, a `VARCHAR` column with check constraint is used — confirm during implementation.

### Indexes

-   `idx_contact_messages_status` on `status` (for filtering)
-   `idx_contact_messages_created_at` on `created_at DESC` (for default sort)

-   Back link to list

-   Full message body (read-only)
-   Status actions:
    -   If `UNREAD`: "Mark as read" button (auto-fires on view)
    -   If `READ`: "Archive" button

-   Delete button with confirmation
-   Loading, not-found, and error states

### Auto-Mark-As-Read Behavior

When the admin opens a detail page for an `UNREAD` message, the frontend should fire a `PUT /status { status: "READ" }` call automatically. This is a UX choice — the backend does not enforce it.

## Suggested Implementation Order

1.  Phase 1 — Database migration
2.  Phase 2 — Entity and Repository
3.  Phase 3 — DTOs and Service
4.  Phase 4 — Controllers
5.  Phase 5 — Backend tests
6.  Phase 6 — Frontend API module
7.  Phase 7 — Routes and navigation
8.  Phase 8 — Admin list page
9.  Phase 9 — Admin detail page

This order is intentional:

-   Data layer first so everything builds on a stable schema.
-   Backend tests before frontend work so the contract is verified.
-   API module before UI pages so the pages can consume a tested client.
-   List before detail because detail is navigated to from the list.
